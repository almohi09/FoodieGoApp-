# Windows Android Build Troubleshooting

This guide documents the recurring Android build failure on Windows where React Native native codegen paths become too long.

## Symptom

Build fails during `:app:buildCMakeDebug[arm64-v8a]` with errors similar to:

- `Filename longer than 260 characters`
- `cannot be safely placed under this directory`
- `react_codegen_safeareacontext` / `react_codegen_rnscreens` ninja/CMake failures

## Root Cause

On Windows, object file paths generated by React Native New Architecture (`safe-area-context`, `screens`, etc.) can exceed `MAX_PATH` when the project directory is deep.

In React Native `0.84.x`, setting `newArchEnabled=false` is not a valid workaround because New Architecture is enabled by default.

## Permanent Project Fix (already applied)

In [android/app/build.gradle](../android/app/build.gradle), keep CMake intermediate output in a short path:

```gradle
externalNativeBuild {
    cmake {
        buildStagingDirectory "C:/fgb_foodiego/cxx"
    }
}
```

This reduces full object path length and prevents the `260 characters` failure.

## Standard Recovery Steps

Run from project root:

```powershell
cd android
.\gradlew.bat clean app:assembleDebug
.\gradlew.bat app:installDebug -PreactNativeDevServerPort=8081
```

Then start Metro and launch:

```powershell
cd ..
adb reverse tcp:8081 tcp:8081
npx react-native run-android --port 8081
```

## If You Removed `@react-native-community/geolocation`

After uninstalling:

```powershell
npm uninstall @react-native-community/geolocation
```

make sure no file still imports it. Use:

```powershell
rg -n "@react-native-community/geolocation|Geolocation" src
```

Use `navigator.geolocation` (or another supported location provider) consistently across services.

## Notes

- Deprecated warnings from dependencies may appear; they are not the cause of this failure.
- If Gradle cache lock errors appear (`.zip.lck Access is denied`), close parallel Gradle runs and retry.
